01 Installer les outils

⬡
Installer Node.js (version 20)
Le moteur JavaScript qui fait tourner Hardhat
À faire

Node.js est nécessaire pour exécuter tous les outils Hardhat. Installez la version LTS 20.

1. Allez sur nodejs.org → cliquez sur le bouton vert "LTS"
2. Lancez le fichier .msi téléchargé, cliquez "Next" partout, laissez toutes les options par défaut
3. Redémarrez votre ordinateur après l'installation

# Option 1 — Homebrew (recommandé si vous l'avez)
brew install node@20

# Option 2 — Téléchargement direct
# Allez sur nodejs.org → téléchargez le .pkg LTS → installez
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

Vérifier l'installation

Ouvrez un terminal (ou PowerShell sur Windows) et tapez :

node --version
# Vous devez voir quelque chose comme :
v20.11.0

npm --version
10.2.4
⚠️ Si la commande n'est pas reconnue Sur Windows, fermez PowerShell et rouvrez-le. Si le problème persiste, redémarrez l'ordinateur.
🌿
Installer Git
L'outil de versioning qui synchronise votre code avec GitHub
À faire

1. Allez sur git-scm.com → cliquez "Download for Windows"
2. Lancez le .exe → laissez toutes les options par défaut (cliquez "Next" partout)
3. À la fin, cochez "Launch Git Bash" pour tester

# Si vous avez Homebrew :
brew install git

# Sinon, Git s'installe automatiquement quand vous tapez :
git --version
# macOS vous proposera d'installer les "Command Line Tools"
sudo apt-get install git

Configurer votre identité Git

Git a besoin de savoir qui vous êtes pour signer vos commits. Faites ça une seule fois :

# Remplacez par vos vraies informations :
git config --global user.name "Prénom Nom"
git config --global user.email "votre@email.com"

Vérifier

git --version
git version 2.44.0

🔑
Connecter Git à votre compte GitHub
Pour pouvoir pousser du code sans retaper votre mot de passe
À faire

Il y a deux méthodes. Prenez la plus simple pour vous :

Méthode A — HTTPS avec token (plus simple)

1. Sur GitHub → cliquez sur votre avatar → Settings
2. Dans le menu gauche → Developer settings → Personal access tokens → Tokens (classic)
3. Cliquez "Generate new token (classic)"
4. Donnez-lui un nom, cochez "repo", cliquez "Generate token"
5. Copiez le token maintenant (vous ne pourrez plus le voir après)

Quand Git vous demande un mot de passe, collez ce token à la place.

Méthode B — SSH (recommandée pour travailler confortablement)

# 1. Générer une clé SSH (appuyez sur Entrée 3 fois)
ssh-keygen -t ed25519 -C "votre@email.com"

# 2. Copier la clé publique
# Sur Mac/Linux :
cat ~/.ssh/id_ed25519.pub
# Sur Windows (PowerShell) :
type $env:USERPROFILE\.ssh\id_ed25519.pub

3. Copiez le texte affiché (commence par ssh-ed25519)
4. Sur GitHub → Settings → SSH and GPG keys → "New SSH key"
5. Collez la clé → cliquez "Add SSH key"

Vérifier la connexion

ssh -T git@github.com
Hi votre-username! You've successfully authenticated.

Télécharge tous les outils nécessaires (~2 min)
À faire

Cette commande lit le fichier package.json et télécharge automatiquement Hardhat, OpenZeppelin, et tous les outils nécessaires dans un dossier node_modules/.

📌 node_modules est dans .gitignore Ce dossier peut peser 200 Mo. C'est pourquoi il est listé dans .gitignore — il ne sera jamais envoyé sur GitHub. Chaque développeur le recrée avec npm install.

06 Sauvegarder son travail avec Git

🔄
Les 3 commandes Git du quotidien
add → commit → push : à faire régulièrement pendant l'atelier
À lire

Pensez à Git comme à un système de points de sauvegarde dans un jeu vidéo. Après chaque avancée significative, sauvegardez :

# Étape 1 — Voir ce qui a changé
git status

# Étape 2 — Sélectionner les fichiers à sauvegarder
git add .          # le point = tous les fichiers modifiés

# Étape 3 — Créer le point de sauvegarde avec un message
git commit -m "feat: complete TODO 1-5 (imports and state variables)"

# Étape 4 — Envoyer vers GitHub
git push
💡 Quand pusher ? Poussez après chaque série de TODO complétés, ou au minimum à la fin de chaque session de travail. Chaque push déclenche la CI automatiquement.

Messages de commit recommandés

git commit -m "feat: complete contract TODO 1-10"
git commit -m "test: add tests T1-T15"
git commit -m "deploy: add contract address to README"
git commit -m "docs: complete NatSpec"

07 Comprendre la CI/CD

🤖
Ce que fait GitHub Actions automatiquement
À chaque push, GitHub exécute vos tests pour vous
À lire

Le fichier .github/workflows/ci.yml est déjà configuré. À chaque fois que vous faites un git push, GitHub :

✓ Installe Node.js et les dépendances
✓ Compile le contrat
✓ Lance tous vos tests
✓ Génère le rapport de coverage
✓ Envoie le résultat à Codecov

Voir les résultats

1. Sur votre repo GitHub → onglet "Actions"
2. Vous voyez la liste de vos pipelines
3. Cliquez sur un pipeline pour voir les détails

github.com/vous/atelier1-audit-token/actions
  ● feat: initial starter kit   ← rouge = tests échouent (normal au début)
  ● feat: complete TODO 1-10     ← jaune = en cours
  ● feat: complete TODO 1-15     ← vert = tous les tests passent ✓

Ajouter le badge CI dans votre README

Une fois que votre CI passe en vert, copiez ce code dans votre README.md (remplacez les valeurs) :

08 Configurer le badge de Coverage

📊
Connecter Codecov à votre repo
Pour afficher le pourcentage de coverage dans votre README
À faire

Codecov est un service gratuit qui lit le rapport de coverage généré par vos tests et affiche un badge dans votre README.

Créer un compte Codecov

1. Allez sur codecov.io
2. Cliquez "Sign up with GitHub" → autorisez Codecov
3. Dans le tableau de bord Codecov → cliquez "Add new repository"
4. Sélectionnez atelier1-audit-token

Copier le token Codecov dans GitHub Secrets

Codecov vous donne un token. Il faut le mettre dans GitHub pour que la CI puisse l'utiliser :

1. Sur Codecov → copiez le token (CODECOV_TOKEN)
2. Sur votre repo GitHub → Settings → Secrets and variables → Actions
3. Cliquez "New repository secret"
4. Name : CODECOV_TOKEN, Value : collez le token
5. Cliquez "Add secret"

💡 Après votre prochain push La CI uploadera automatiquement le rapport de coverage vers Codecov. Le badge s'activera dans les minutes qui suivent.

09 Problèmes fréquents

❌ "npm : command not found" ou "npm n'est pas reconnu" ▼
Node.js n'est pas installé ou pas dans le PATH. Solutions :

1. Vérifiez que vous avez bien installé Node.js (étape 1)
2. Fermez et rouvrez votre terminal
3. Sur Windows : redémarrez l'ordinateur
4. Sur Mac : relancez VS Code après l'installation de Node
❌ "git push" demande un mot de passe en boucle ▼
Vous utilisez HTTPS mais n'avez pas de token configuré. Deux solutions :

Solution rapide : quand Git demande le mot de passe, collez votre Personal Access Token (pas votre mot de passe GitHub).

Solution durable : configurez la méthode SSH (voir l'étape "Connecter Git à GitHub").
❌ La CI est rouge et affiche "Compilation failed" ▼
Normal tant que le contrat n'est pas complet ! La CI devient verte une fois que :

1. Tous les TODO du contrat sont implémentés
2. Le contrat compile sans erreur (npm run compile → 0 erreurs)
3. Tous les tests passent (npm test → tous verts)
❌ "ENOENT: .env file not found" lors du déploiement ▼
Vous n'avez pas créé le fichier .env. Faites :

cp .env.example .env

Puis ouvrez .env et remplissez les variables (PRIVATE_KEY, SEPOLIA_RPC_URL, ETHERSCAN_API_KEY).
❌ "Insufficient funds" lors du déploiement ▼
Votre compte de test n'a pas assez d'ETH Sepolia. Obtenez-en gratuitement sur :

• sepoliafaucet.com
• faucets.chain.link/sepolia

Attendez 1-2 minutes après la réception avant de retenter.
❌ Le badge CI reste gris après le push ▼
Le badge peut mettre 1-2 minutes à s'afficher.

Si après 5 minutes il est toujours gris :
1. Vérifiez que votre repo est bien Public
2. Allez dans l'onglet Actions de votre repo et vérifiez qu'un workflow a bien tourné
3. Copiez l'URL du badge depuis Actions → votre workflow → le bouton "···" → "Create status badge"
❌ "Already Verified" lors du déploiement ▼
Ce n'est pas une erreur. Cela signifie que votre contrat est déjà vérifié sur Etherscan. Vous pouvez ignorer ce message.
🤔 Je ne comprends pas un TODO — comment avoir de l'aide ? ▼
Dans l'ordre :

1. Relisez l'indice dans le fichier .sol ou .test.js
2. Ouvrez le guide interactif (atelier1-apprenant.html) — chaque TODO y est expliqué
3. Cherchez sur la doc OpenZeppelin : docs.openzeppelin.com
4. Cherchez sur la doc Hardhat : hardhat.org/docs
5. En dernier recours, posez votre question avec le message d'erreur exact
